## 3. Discord Bot のパーミッション要件

### 🔐 Discord Developer Portal での設定

**Step 1: アプリケーション作成**

1. [Discord Developer Portal](https://discord.com/developers/applications) にアクセス
2. `New Application` → アプリ名を入力（例: `OmikujiBot`）
3. `Bot` タブ → `Add Bot`

**Step 2: 必要な権限設定**

```
【今回のBotに必要な権限】

Bot Permissions（OAuth2 → URL Generator で選択）:
  ✅ Send Messages           （メッセージ送信）
  ✅ Use Slash Commands       （スラッシュコマンド）
  ✅ Read Message History     （履歴読み取り、エラー解析用）
  ✅ Embed Links             （埋め込みカード送信）
  ✅ Add Reactions            （リアクション追加）
  ✅ Attach Files             （ファイル添付）※拡張用

Privileged Gateway Intents（Botタブ下部）:
  ✅ Message Content Intent  （メッセージ本文読み取り）
  ✅ Server Members Intent   （メンバー情報取得）
  ✅ Presence Intent         （存在状態）※将来の拡張用
```

> ⚠️ **注意:** `Message Content Intent`は Discordの規制強化後、**明示的にONにしないとメッセージが読めない**。Botが反応しない原因の第1位がこれ。

**Step 3: Token の取得と管理**

```
Bot タブ → Reset Token → コピー

【絶対にやってはいけないこと】
❌ TOKEN を GitHub にプッシュする
❌ TOKEN をコードにべた書きする
❌ TOKEN を Discord のチャットに貼る

【正しい管理方法】
✅ .env ファイルに書く
✅ .gitignore に .env を追加する
✅ GCP では Secret Manager を使う（Part 4 で解説）
```

**.env ファイルの例:**

```env
DISCORD_TOKEN=your_bot_token_here
CLIENT_ID=your_client_id_here
GUILD_ID=your_test_server_id_here
```

**.gitignore に必ず追加:**

```gitignore
node_modules/
.env
.env.local
dist/
*.log
```

## 5. IDE（エディタ）選定の思想 — これが一番意見が分かれる

### 🏛️ IDE選定は「哲学の問題」

IDEの選び方は、2026年において最も意見が分かれるトピックだ。  
「Cursor一択」「いやVSCodeで十分」「Windsurfが最強」「Claude Codeがあれば何もいらない」——  
どれも間違いではなく、**開発スタイルと目的によって最適解が変わる**。

ここでは、主要IDEを「哲学の違い」で整理する。

### 📐 IDE の3つの流派

```
【流派1: 伝統派】 VSCode (+ AI拡張)
  「自分がドライバー。AIは助手席」
  
  → AIはあくまで補助。コードは自分が書く。
  → 拡張機能の自由度が最高。
  → Gemini Code Assist / Copilot / Cline 等を組み合わせて使う。
  → 「自分の手でコードを書いて理解したい」人向け。

【流派2: AIネイティブ派】 Cursor / Windsurf / Antigravity
  「AIと並走する。ドライバーはAIかもしれない」
  
  → エディタ自体がAIを中心に設計されている。
  → マルチファイル編集・コードベース全体の把握が強み。
  → 「AIに任せて、自分は設計に集中したい」人向け。

【流派3: 委任派】 Claude Code (CLI)
  「AIに全部やらせる。自分はディレクター」
  
  → ターミナルから指示を出すと、AIがファイルを読み書きし実行まで行う。
  → エディタはほぼ使わない（確認用に使う程度）。
  → 「大きな変更・リファクタリングをまるごと委任したい」人向け。
```

### 🔍 主要IDE詳細比較（2026年3月時点）

#### VSCode（流派1の代表）

```

  - 「AI拡張を自分で選べる」柔軟性

❌ 短所
  - AIネイティブではないので、マルチファイル編集は手間
  - 「AIに全部やらせる」には向かない

🎯 向いている人
  - コードを自分で書いて理解したい
  - 無料で戦いたい
  - 既存のVSCode環境を維持したい（本講座の筆者もここ）
```

#### Cursor（流派2の代表・バイブコーディング御用達）

```

🔧 AIモデル: Claude Sonnet 4.6 / GPT-5.2 等を切り替え可能

✅ 長所
  - Composer（Agent Mode）によるマルチファイル編集が秀逸
  - 「Ctrl+I でタスクを書いて委任する」体験が最高に気持ちいい
```

#### Windsurf（流派2・Cascade が個性的）

```
💰 価格: 無料枠あり / Pro $15/月（Cursorより安い）
🔧 AIモデル: 独自モデル（SWE-1.5）+ 外部モデル切り替え可

✅ 長所
  - Cascade（エージェントシステム）がセッション内で文脈を保持

  - コミュニティ情報がCursorより少ない

🎯 向いている人

  - AIに「状況を覚えていてほしい」（Flowsモデル）
  - コストを抑えたい（$15/月）
```


### 🏆 筆者推奨：二刀流構成

```
VSCode + Gemini Code Assist（Antigravity）

・Gemini Code Assist: 無料枠が厚い・Googleのモデルなので精度も十分
・この2つを「当たり前の環境」として手に馴染ませると、
  AIとのペアプログラミングが自然体になる

【補助輪（無料枠の範囲で回す）】
ブラウザタブ常駐:
  - Claude.ai (重い処理・ZIPやり取り用)
  - ChatGPT (アイデア出し・エラー分析)
  - Gemini.google.com (プロンプト整形・Google連携)
```

## 7. バイブコーディングの思想とワークフロー

### 🎯 バイブコーディングとは何か

「バイブコーディング（Vibe Coding）」とは、2025〜2026年に広まった開発スタイルで、**コードを書く前に思考とアイデアをAIと対話で研ぎ澄ませる**アプローチだ。

従来型:
```
要件定義 → 設計 → コーディング → テスト → デバッグ
（全部自分でやる。辛い。）
```

バイブコーディング型:
```
思想・雰囲気・要件をAIに投げる
   ↓
AIとの対話でアイデアを発散・収束させる
   ↓
「これをコードにして」と委任する
   ↓
動かなければAIに投げる
   ↓
良くなったら次の機能へ
（自分は「何を作るか」に集中する。AIは「どう作るか」を担当）
```

### 🔄 筆者の実際のワークフロー（本講座のモデル）

```
Step 1: 思想フェーズ（ChatGPT）
  「Discord Botでおみくじを作りたい。どんな機能があると面白い？」
  → ChatGPT と雑談感覚でアイデアを広げる
  → 10個以上のアイデアが出てくる
  
Step 2: ブラッシュアップ（Gemini）
  「↑のアイデアをGeminiにも見せて、追加で何か面白いこと言える？」
  → Gemini視点の提案が加わる
  → 競合させることで死角がなくなる

Step 3: プロンプト設計（Gemini）
  「この要件をもとに、Claudeへの指示プロンプトを作って」
  → Geminiが「Claudeに最適化したプロンプト」を作ってくれる
  → これが職人技。Geminiはプロンプト整形が得意。

Step 4: 実装（Claude）
  「↑のプロンプトをClaudeに投げる」
  → ZIPファイルで受け取る
  → ファイル構成まで整っている

Step 5: テスト・デバッグ（ChatGPT or Claude）
  うまく動かない → ZIPをChatGPTに投げて原因分析
  複雑なバグ    → ZIPをClaudeに再度投げて修正依頼

Step 6: 新機能追加（Claude + Codex/Cursor）
  Claude でアーキテクチャ変更
  Copilot / Cursor で細部の実装・補完
```

> **これが2026年の「最高のバイブコーディングエンジニア」のやり方だ。**  
> コードを書く量を最小化して、**思考と設計に時間を使う**。
> AIはあなたの手を動かす代理人であり、あなたは「何を作るか」の意思決定者だ。

## 8. 課金するならどのAIか / 課金しないコツ

### 💰 課金するなら優先順位

```
第1位: Claude Pro ($20/月)
  理由: Claude Code が使える。ZIPやり取りの精度が圧倒的。
        Claude Sonnet 4.6 の長いコンテキストでプロジェクト全体を渡せる。
        コードの品質が最も安定している。

第2位: ChatGPT Plus ($20/月)
  理由: GPT-5.2 / o3 の制限が緩和される。
        o3（推論モデル）で複雑なロジックを解かせると精度が上がる。
        ファイルアップロード・Canvas機能が使いやすい。
        アイデア出しの量が増える。

第3位: GitHub Copilot ($10/月)
  理由: エディタ常駐で摩擦ゼロ。無料枠が2024年末に拡張されたが、
        本格利用には課金が必要。
        毎日コードを書く人なら元が取れる。

→ 予算が月$20なら迷わずClaude Pro。
→ 月$0で頑張るなら Gemini Code Assist + ChatGPT Free + Claude Free の三角形。
```

### 🆓 無料で戦い続けるコツ

```
【AIの消費を最適化する技術】

1. 「重い質問」を朝イチに回す
   → 無料枠のリセットタイミングを把握する
   → ChatGPT: 3時間でリセット、Claude: 概ね1日でリセット

2. 質問を一発で終わらせる技術
   → 「5W1H + 期待するアウトプット」を最初のメッセージに全部入れる
   → 往復回数を減らすことで消費量を減らす

3. AIを使い分けてリセット待ちを回避
   → ChatGPTが制限 → Claudeに移行 → Geminiで補完
   → 3サービス同時並行で実質的に制限を感じにくくする

4. Gemini Code Assist をベースにする
   → エディタの補完はほぼ無制限（2026年時点）
   → 「軽い質問」はGemini Code Assistに任せる

5. Google AI Studio を使う
   → Gemini APIを直接叩けるPlayground
   → プロンプト実験・テストはここで行う（無料枠あり）
   → https://aistudio.google.com/

6. ZIPを大切に使う（Claude活用法）
   → Claudeに全ファイルを渡すときはZIPにまとめる
   → 「このZIPのbot.jsのここを直して」と具体的に指定する
   → 曖昧な指示 = 複数回の往復 = 消費増加
```

## 9. デバッグはどのAIが優れているか

### 🐛 デバッグAI格付け（2026年3月時点）

```
【第1位】Claude（エラーの根本原因追跡）
  強み: 複数ファイルにまたがるバグの因果関係を把握する能力が高い
  使い方: エラーログ + 関連ファイルをまとめてZIPで渡す
  「このエラーが出ています。原因を特定して修正してください。」
  → ファイル構成ごと直したZIPを返してくれる

【第2位】ChatGPT o3（論理的なデバッグ・推論）
  強み: 推論モデルなので「なぜこのエラーが起きるか」の仮説が深い
        複雑なロジックバグや非同期処理の問題に強い
  使い方: エラーと関連コードを貼り付けて「step by stepで原因を分析して」

【第3位】ChatGPT GPT-5.2（エラーメッセージの解読・仮説出し）
  強み: スタックトレースを読ませると的確な原因候補を出す
        「よくあるミス」のデータベースが豊富。速い。
  使い方: エラーメッセージのみを貼り付けて「原因と解決策を教えて」

【第4位】Gemini Code Assist（エディタ内リアルタイム）
  強み: コードを書きながらリアルタイムに「ここおかしくない？」が分かる
        エディタ内で質問できるので摩擦が最小
  使い方: エラーが出た行を選択 → 右クリック → "Gemini: Explain"
```

### 🔍 デバッグワークフロー実例

```
エラー発生
   ↓
Step 1: Gemini Code Assist でエラー行を確認（30秒）
   ↓（解決しない場合）
Step 2: ChatGPT にエラーログを貼り付けて原因仮説を出す（2分）
   ↓（複雑な場合）
Step 3: Claude にZIPを渡して「完全に直して」（5〜10分）
   ↓
完了
```
